## Executable `/goal` body

实现 MikuCode B 版 Task 2：运行时数据模式（AgentAction / ToolResult / AgentState）、事件（AgentEvent）、以及带 secret redaction 的 JSONL Trace Recorder。严格按 docs/superpowers/plans/2026-07-07-mikucode-implementation.md 的 Task 2 执行，不得提前实现 Task 3+ 功能（尤其不得实现 ToolRegistry 与 AgentRuntime loop）。

### 1. 目标（What / Why）

在 Task 1 已落地的 `mikucode` 包之上，建立 agent 运行时的“数据语言”层：

- `AgentAction`：模型输出的归一化动作协议，5 种 type（`tool_call` / `patch_proposal` / `plan_update` / `ask_user` / `final_answer`），用 **pydantic BaseModel + model_validator** 按 type 做字段校验
- `ToolResult`：工具执行结果的统一结构（ok/tool/summary/content/metadata）
- `AgentState`：会话状态（task/step_count/observations/files_read/files_modified/verification_state），含 `new` / `record_observation` / `next_step`
- `AgentEvent`：trace 事件结构（type/step/timestamp/payload）
- `TraceRecorder`：把 `AgentEvent` 追加写入 `.miku/sessions/<timestamp>-session.jsonl`，**写入前必须做 secret redaction**
- `redact_secrets(text)`：独立可测的脱敏函数

本 Task 是后续工具执行（Task 3）、patch/verification（Task 4+）的契约地基：动作协议、状态机语义、trace 落盘格式必须先稳定，且 agent 决策 loop 本身**不在本 Task**实现。

### 2. 范围（In Scope）

仅创建/修改以下文件（与计划一致）：

- `src/mikucode/runtime/__init__.py`（空/极简，仅使包可导入）
- `src/mikucode/runtime/actions.py`
- `src/mikucode/runtime/events.py`
- `src/mikucode/runtime/state.py`
- `src/mikucode/tracing/__init__.py`（空/极简，仅使包可导入）
- `src/mikucode/tracing/recorder.py`
- `tests/test_runtime_schemas_and_tracing.py`

接口契约（必须实现且可被后续 Task import，签名冻结）：

- `AgentAction(BaseModel)`：
  - 字段 `type: Literal["tool_call","patch_proposal","plan_update","ask_user","final_answer"]`，以及 `tool / arguments / patches / reason / summary / changed_files / verification / remaining_risks / items / question / options / risk_level`（各字段默认值见 plan）
  - `@model_validator(mode="after")` 校验：`tool_call` 必须有 `tool`；`patch_proposal` 必须有非空 `patches`；`ask_user` 必须有 `question`；`final_answer` 必须有 `summary`
  - 支持 `AgentAction.model_validate(dict)` 从原始 JSON dict 构造
- `ToolResult(BaseModel)`：`ok: bool`, `tool: str`, `summary: str`, `content: str = ""`, `metadata: dict[str, Any] = {}`
- `AgentState(BaseModel)`：`task / step_count=0 / done=False / observations=[] / files_read=set / files_modified=set / verification_state="unknown"`；classmethod `new(task)`；`next_step()` 使 `step_count += 1`；`record_observation(result)` 仅 append 到 `observations`
- `AgentEvent(BaseModel)`：`type: str`, `step: int = 0`, `timestamp: str`（默认 `datetime.now(UTC).isoformat()`）, `payload: dict = {}`
- `TraceRecorder(miku_dir: Path)`：构造时确保 `miku_dir/sessions/` 存在，并以 UTC 时间戳生成 `self.path = sessions/<YYYYmmddTHHMMSSZ>-session.jsonl`
- `TraceRecorder.record(event: AgentEvent) -> None`：序列化为 JSON 后 **先 redact 再追加写入**（每行一个事件，行尾 `\n`）
- `redact_secrets(text: str) -> str`：按 plan 给出的正则集合脱敏，`Authorization: Bearer ...` -> `Authorization: Bearer [REDACTED]`，其余 `KEY=value` 形态 -> `KEY=[REDACTED]`，裸 `sk-...` -> `[REDACTED]`

消费：`MikuConfig.miku_dir`（来自 Task 1，已存在）作为 `TraceRecorder` 的入参根目录。

### 7. 设计与代码质量闸门

- **pydantic 而非 dataclass**：`AgentAction` / `ToolResult` / `AgentState` / `AgentEvent` 全部用 `pydantic.BaseModel`；`AgentAction` 的 type 校验用 `@model_validator(mode="after")`，**禁止**用 dataclass + 手写 `__post_init__` 替代
- **校验错误信息可断言**：缺字段异常信息必须包含缺失字段名（如 `"tool is required for tool_call"`），以支撑 `"tool" in str(exc)` 风格断言
- **状态机语义最小且正确**：`record_observation` 只记观察、不动步数；`next_step` 才推进步数。这是后续 agent loop 的不变量，不得在本 Task 偷偷耦合
- **trace 先脱敏后落盘**：`record` 内顺序必须是 `serialize -> redact -> write`；禁止“先写明文再脱敏”或“只脱敏 payload 不脱敏整行”
- **幂等安全**：`TraceRecorder.__init__` 的 mkdir 用 `exist_ok=True`；多次实例化不报错
- **路径与时区**：trace 文件名用 UTC，避免本地时区漂移；路径全程 `pathlib.Path`
- **类型与现代 Python**：使用 `str | None`、`dict[str, Any]`、`list[...]`、`set[str]` 等 3.11+ 语法；`datetime.now(UTC)`
- **测试隔离**：只用 `tmp_path`，不污染真实仓库 `.miku/`
- **不要**为了“更完美”重写计划中的 API 形状；后续 Task 依赖这些名字与签名

### 8. 已知坑与对抗式检查（必须自检）

执行中主动检查并规避：

1. **误用 dataclass**：`AgentAction` 必须是 pydantic `BaseModel`；若写成 dataclass，`model_validate` 与 `model_validator` 都不存在，测试会全红
2. **model_validator 模式**：必须 `mode="after"`，在字段已解析后做跨字段校验；`mode="before"` 拿不到 `self.tool` 等
3. **校验错误信息措辞**：测试用 `"tool" in str(exc)`；若抛 `"missing field"` 但不含 `"tool"`，用例 2 会假失败。pydantic v2 的 `ValidationError` 继承自 `ValueError`，`model_validate` 抛出可被 `except ValueError` 捕获——确认行为符合
4. **step_count 语义**：`record_observation` 后 `step_count == 0` 是**有意**的；不要“顺手”在 `record_observation` 里 `next_step()`。步数推进是 agent loop 的职责
5. **redact 顺序依赖**：`OPENAI_API_KEY=\S+` 必须在 `sk-[A-Za-z0-9_-]+` 之前匹配，否则 `sk-secret` 会被裸 sk 模式先吃掉，导致整行变成 `OPENAI_API_KEY=[REDACTED]` 而非 `OPENAI_API_KEY=[REDACTED]`（结果巧合相同，但语义错）。保持 plan 给定的 pattern 顺序
6. **Bearer 替换的精确输出**：测试要求严格等于 `Authorization: Bearer [REDACTED]`（含 `Bearer `）；若只输出 `Authorization: [REDACTED]` 会失败。注意 plan 中 `pattern.pattern.startswith("Authorization")` 分支写死该字符串
7. **redact 作用于整行 JSON**：`event.model_dump_json()` 输出单行 JSON；secret 出现在 payload 值里、未被 JSON 转义（`sk-secret`、`=`、空格均不转义），正则可命中。但若 payload 含会被 JSON 转义的字符（如 `\`、`"`），正则可能漏匹配——本 Task 测试范围内不触发，但需知晓边界
8. **sk- 模式过度匹配**：`sk-[A-Za-z0-9_-]+` 会误伤任何 `sk-` 开头串（如 `sk-123`）。本 Task 可接受，但不要把 redact 当作安全唯一防线
9. **set 字段可变默认**：`files_read` / `files_modified` 用 `Field(default_factory=set)`，禁止用类属性共享可变默认
10. **遗漏 `__init__.py`**：缺 `runtime/__init__.py` / `tracing/__init__.py` 会导致 `import mikucode.runtime.actions` 失败，红灯变 ImportError 而非断言失败——不影响红灯判定，但实现时必须补上
11. **Windows 编码/换行**：写文件统一 `encoding="utf-8"`，行尾 `\n`；`read_text(encoding="utf-8")` 读回
12. **超范围铺路**：禁止创建 `tools/`、`agent.py`、`runtime/loop.py` 等 Task 3/4 文件，哪怕只是空 stub

若测试失败：先读失败断言与 traceback，修最小集合，禁止用删测试/改断言方式“过关”，除非断言与计划冲突（冲突时以计划为准并在总结中说明）。

## Design notes (meta)

Why this goal is written this way:

1. **Lock the pydantic choice up front** - agents may reach for dataclass; the DoD + known-pitfalls block that explicitly.
2. **Freeze the step_count invariant** - `record_observation` must not bump steps; this is a load-bearing semantic for the Task 4 agent loop, so it is promoted to a DoD row and a known-pit.
3. **Redact-before-write is non-negotiable** - the design spec lists "trace leaks secrets" as a top risk; the goal mandates serialize -> redact -> write ordering and a red-light test asserting no plaintext `sk-secret` on disk.
4. **API contract is frozen** - Task 3 (ToolRegistry) consumes `ToolResult`; Task 4 (agent loop) consumes `AgentAction`/`AgentState`/`TraceRecorder`; signature drift here is expensive.
5. **No loop, no registry** - Out of Scope explicitly forbids Task 3/4 surface area to prevent pre-building.
6. **Acceptance is machine-checkable** - commands, file presence, and exact redaction string equality, not "looks done."
