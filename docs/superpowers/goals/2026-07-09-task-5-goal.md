## Executable `/goal` body

实现 MikuCode B 版 Task 5：Patch Engine、Diff Preview、Backup、Undo 和 Patch Permissions。严格按 docs/superpowers/plans/2026-07-07-mikucode-implementation.md 的 Task 5 执行，不得提前实现 Task 6+ 功能。

### 1. 目标（What / Why）

在仓库 A:\study\AI\LLM\Fable-5 中，为 MikuCode 添加基于 patch proposal 的安全编辑能力：

- `SearchReplacePatch`：old_text 必须在文件中唯一匹配才允许替换，防止误改
- `CreateFilePatch`：目标文件不存在时才允许创建（`if_exists=fail`），防止覆盖
- `PatchEngine.apply_patches`：批量应用 patch，每个 patch 先备份再写入，返回 unified diff
- `UndoManager.undo_last`：读取最近一次备份清单，恢复所有被修改的文件
- `DiffEngine.unified_diff`：生成 before/after 的 unified diff 文本
- `runtime/agent.py` 新增 `patch_proposal` 分支：将 `AgentAction.patches` 交给 PatchEngine 执行

本 Task 是 MikuCode "模型提议、运行时验证执行" 编辑模型的核心：所有文件修改必须走 patch 流程，禁止模型直接 write_file。

### 7. 设计与代码质量闸门

- **唯一匹配约束**：`search_replace` 的 `old_text` 必须在文件中出现恰好 1 次；0 次或多次均拒绝。这是防止模型误改的核心安全机制
- **备份先于写入**：`create_backup` 必须在 `write_text` 之前调用；若备份失败则中止 patch
- **PathPolicy 复用**：PatchEngine 必须使用 Task 3 的 `PathPolicy.ensure_inside_project` 校验路径，防止路径逃逸
- **幂等安全**：`create_file` 在文件已存在时 fail（`if_exists=fail`），不覆盖
- **原子性**：`apply_patches` 中任一 patch 失败则整体中止，不部分应用
- **类型与现代 Python**：使用 `Path`、`dataclass(frozen=True)`、`str | None` 等 3.11+ 语法
- **测试隔离**：只用 `tmp_path`，不污染真实仓库
- **不要**为了"更完美"重写计划中的 API 形状；后续 Task 依赖这些名字与签名

### 8. 已知坑与对抗式检查（必须自检）

执行中主动检查并规避：

1. **old_text 含尾随换行**：`read_text()` 读取的文件内容含尾随 `\n`，`old_text` 参数可能不含，导致 count=0。需确保匹配逻辑对换行敏感
2. **备份覆盖**：多次 patch 的 `last.json` 只记录最后一次；undo 只能恢复最近一次。这是 B 版本的已知限制，不是 bug
3. **create_backup 的 patch_id 碰撞**：同一秒内两次 patch 会生成相同 patch_id。B 版本可接受此限制
4. **PathPolicy 对 .miku 路径**：备份写入 `.miku/backups/` 不应被 PathPolicy 拦截；PatchEngine 的 PathPolicy 仅用于校验用户文件路径
5. **diff 空文件**：`create_file` 的 before 为空字符串，`unified_diff` 需正确处理全新增的情况
6. **Windows 行尾**：`difflib.unified_diff` 使用 `splitlines(keepends=True)` 保留行尾；写文件统一 `encoding="utf-8"`
7. **undo 后 last.json 未清理**：undo 后再次 undo 会重复恢复同一备份。B 版本可接受
8. **patch_proposal 的 patches 为空**：`AgentAction` 的 model_validator 已要求 `patches` 非空，但 PatchEngine 也应防御空列表

若测试失败：先读失败断言与 traceback，修最小集合，禁止用删测试/改断言方式"过关"，除非断言与计划冲突（冲突时以计划为准并在总结中说明）。
