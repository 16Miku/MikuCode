## Executable `/goal` body

实现 MikuCode B 版 Task 4：JSON Action Parser、Mock Provider 和 Agent Runtime Loop。严格按 docs/superpowers/plans/2026-07-07-mikucode-implementation.md 的 Task 4 执行，不得提前实现 Task 5+ 功能。

### 1. 目标（What / Why）

在仓库 A:\study\AI\LLM\Fable-5 中，为 MikuCode 落地"模型决策、运行时执行"的核心循环：

- `ModelProvider` Protocol：定义 `complete(messages, tools=None) -> ModelResponse` 的统一接口，后续 Task 8 的 OpenAI-compatible provider 与本 Task 的 MockProvider 都实现它
- `MockProvider(responses: list[str])`：按预设 JSON 字符串序列返回 `ModelResponse`，让 runtime loop 可被确定性测试
- `AgentRuntime(project_root, provider, registry, max_steps).run(task) -> AgentState`：核心循环，把模型的 JSON 输出解析为 `AgentAction`，按 action type 分发执行，并把每一步写入 trace
- `ContextBuilder`（最简版）：仅组装 task + recent observations，让循环可被测试驱动；完整版留给 Task 7

本 Task 是 MikuCode 的中枢：把 Task 2 的 schema/state/trace、Task 3 的 tool registry 串成一个可运行的 agent。循环骨架定型后，Task 5 的 patch_proposal 与 Task 7 的完整 context 都是在此骨架上加分支。

#### Step D - 实现 AgentRuntime 循环

创建 `src/mikucode/runtime/agent.py`，从 Task 2 导入 `AgentAction`, `ToolResult`, `AgentState`, `AgentEvent`, `TraceRecorder`，从 Task 3 导入 `ToolRegistry`：

- `AgentRuntime.__init__(project_root, provider, registry, max_steps=20)`：resolve root；`self.recorder = TraceRecorder(ensure_miku_dir(self.project_root))`
- `run(task: str) -> AgentState` 核心循环顺序：
  1. `state = AgentState.new(task)`；记录 `session_started` 事件
  2. 循环 `while not state.done`：
     - **max_steps 检查**：若 `state.step_count >= self.max_steps`，构造 `ToolResult(ok=False, tool="runtime", summary="Stopped because max_steps was reached")`，`state.record_observation(result)`，记录 `session_stopped` 事件，`return state`（`done` 保持 False，记录 partial report，**不**假装完成）
     - **build_context**：`messages = self._build_messages(state)`
     - **call_model**：`response = self.provider.complete(messages=messages, tools=None)`
     - **parse_action**：`action = self._parse_action(response.content)`
     - `state.next_step()`；记录 `action_parsed` 事件
     - **validate / dispatch** by `action.type`：
       - `final_answer` -> `state.done = True`；记录 `final_report` 事件；`return state`
       - `tool_call` -> `result = self.registry.execute(action.tool or "", action.arguments)`；`state.record_observation(result)`；记录 `tool_result` 事件；`continue`
       - 其他 type（`patch_proposal` / `plan_update` / `ask_user`）-> 构造 `ToolResult(ok=False, tool="runtime", summary=f"Unsupported action type in this task: {action.type}")`；`state.record_observation(result)`；记录 `validation_failed` 事件；`continue`
  3. 循环出口 `return state`
- `_build_messages(state)`（ContextBuilder 最简版）：返回三条消息——system 提示 "Return exactly one JSON AgentAction."；user 即 `state.task`；system 为 `json.dumps({"recent_observations": [obs.model_dump() for obs in state.observations[-5:]]})`
- `_parse_action(content)`：`json.loads` 解析；`json.JSONDecodeError` 包成 `ValueError(f"Model returned invalid JSON: {content}")`；再用 `AgentAction.model_validate(data)` 校验

注意：本 Task 的 `_build_messages` 是 ContextBuilder 的最简版，Task 7 会用完整 `ContextBuilder.build(state)` 替换它。保持方法名 `_build_messages`，便于 Task 7 只替换方法体。

### 7. 设计与代码质量闸门

- **边界清晰**：`models/` 只做 provider 接口与实现；`runtime/agent.py` 只做循环与解析；不把工具执行、patch、context 全量逻辑塞进 agent
- **核心循环顺序固定**：`build_context -> call_model -> parse_action -> validate -> check_permission -> execute_tool -> record -> continue/final`。本 Task 暂无 `check_permission` 分支（permission 在工具内部由 PathPolicy/risk classifier 处理），但循环结构须为后续插入留位
- **两种 action type 限定**：本 Task 仅处理 `tool_call` 与 `final_answer`；其余 type 一律走 `validation_failed` 分支，禁止提前实现
- **max_steps 语义**：默认 `20`；超限时记录 partial report（`session_stopped`），`done` 保持 `False`，**禁止**把超时伪装成成功完成
- **trace 必须可重放**：每个关键节点（`session_started` / `action_parsed` / `tool_result` / `final_report` / `session_stopped` / `validation_failed`）都 `recorder.record`，事件 `step` 字段对齐 `state.step_count`
- **类型与现代 Python**：使用 `Path`、`Protocol`、`@dataclass(frozen=True)`、`list[dict]` 等 3.11+ 语法
- **测试隔离**：只用 `tmp_path`，不污染真实仓库；每个用例独立 `ToolRegistry` 与 `MockProvider`
- **不要**为了"更完美"重写计划中的 API 形状；后续 Task 依赖这些名字与签名（`ModelProvider.complete` / `AgentRuntime.run` / `_build_messages`）

### 8. 已知坑与对抗式检查（必须自检）

执行中主动检查并规避：

1. **JSON 解析失败**：模型返回非 JSON（或带 markdown 代码块包裹）时 `json.loads` 抛 `JSONDecodeError`。本 Task 必须包成 `ValueError` 并保留原始内容用于诊断。**不要**在本 Task 实现宽松解析（剥离 ```json fence 等），保持严格；真实 provider 的清洗留给 Task 8
2. **max_steps 死循环**：`while not state.done` 必须在循环顶部先检查 `step_count >= max_steps` 再调模型，否则会多走一步。`step_count` 初始为 0，`max_steps=1` 时应只允许 0 次 call_model 后即停（确认与 `test_runtime_stops_at_max_steps` 断言 `step_count == 1` 一致——即触发一次 stop 检查前已 `next_step` 一次，或检查点设在 next_step 之前，需对齐计划示例的语义：`step_count == 1` 表示已执行 1 步后达到上限）
3. **trace 写入失败**：`TraceRecorder` 需 `sessions/` 目录存在；`AgentRuntime.__init__` 已通过 `ensure_miku_dir` 保证。若 `tmp_path` 无写权限会失败——测试用 `tmp_path` 即可
4. **MockProvider index 越界**：测试若 responses 条数不足循环次数，`complete` 必须返回兜底 `final_answer` 而非 `IndexError`，否则循环无法收敛。兜底 summary 文案需与计划一致 `"No more mock responses."`
5. **`action.tool` 为 None**：`tool_call` 分支取 `action.tool or ""` 传给 registry；registry 对未知 tool 返回 `ok=False` 的 `ToolResult`，不会抛异常。确认 `ToolRegistry.execute` 对未知 name 的行为与 Task 3 一致
6. **`state.done` 与 `final_answer`**：只有 `final_answer` 才置 `done=True`；max_steps 超限与 unsupported type 都不得置 done
7. **trace 文件名时间戳碰撞**：同一秒内多次 `AgentRuntime.__init__` 会生成同名 session 文件并 append。B 版本可接受（同会话追加），但跨用例需用独立 `tmp_path` 避免串扰
8. **`_build_messages` 的 observations 序列化**：`ToolResult` 含 `dict`/`set` 之外都是基础类型，`model_dump()` 后 `json.dumps` 安全；但若后续 state 含 `set`（如 `files_modified`），**不要**把整个 state 塞进消息，只取 `observations[-5:]`
9. **循环引用**：`runtime/agent.py` 导入 `models.base`、`tools.registry`、`tracing.recorder`、`runtime.*`、`config`；确认无反向导入导致 ImportError
10. **越界实现 patch_proposal**：对抗式自检——若手痒写了 `if action.type == "patch_proposal"` 分支，必须删除，留给 Task 5

若测试失败：先读失败断言与 traceback，修最小集合，禁止用删测试/改断言方式"过关"，除非断言与计划冲突（冲突时以计划为准并在总结中说明）。

## Design notes (meta)

Why this goal is written this way:

1. **Freeze the loop shape** - `build_context -> call_model -> parse_action -> validate -> check_permission -> execute_tool -> record -> continue/final` is the spine every later Task extends; pinning it now prevents rewrites.
2. **Two action types only** - allowing `patch_proposal`/`plan_update`/`ask_user` to leak in early breaks Task 5/7 sequencing; the `validation_failed` fallback keeps the loop safe without implementing future behavior.
3. **max_steps must not fake success** - `done=False` + `session_stopped` partial report is a hard contract; agents that mask timeouts as completion are the classic agent bug.
4. **CLI deferred to Task 8** - the plan lists `cli/main.py` under Task 4 files, but the commit and the actual wiring both belong to Task 8; stating this explicitly avoids a scope-creep edit.
5. **MockProvider boundary safety** - index-out-of-range returning a fallback `final_answer` (not raising) is what makes the loop convergent under underspecified test fixtures.
6. **Strict JSON parsing** - no fence stripping in Task 4; real-provider cleaning is Task 8's job. Keeping parsing strict makes parse failures visible in tests.
